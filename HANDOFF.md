## What this project is

FaCoS is a Node-RED flow on one tab (id `ae42528ac326c82b`) that emulates
an ESU ECoS command station over TCP/15471, speaking the PC-Interface
protocol (ESU spec v0.2, §7) plus the JMRI / Rocrail superset. Real
throttles (iTrain, JMRI, Rocrail) connect to it as if to ECoS hardware;
the parser translates their commands into Home Assistant calls and MQTT
publishes against the user's existing layout, and translates HA
binary-sensor state back into `<EVENT 200>` feedback frames. The
deliverable is `flow.json` — paste-importable onto any Node-RED instance
that already has the matching MQTT broker and HA server config nodes (or
edit `layout-map.json` / `connection.json` to point at a different host).

## Known coverage gaps (watch the trace log for these)

These are the four most likely command shapes the new parser will mishandle
the first time a real throttle hits it. Each one's symptom and the
file:line where the fix would go.

All four are flagged from a code audit, not from a live capture — there
is no live capture yet (Layer 6 hasn't been run). They are "expect
these in `cursor-scratch/facos-trace.log` or the `unhandled` debug
node" candidates, not confirmed bugs.

1. **`queryObjects(11, name1)` / `addr` / `addrext`** (JMRI accessory
   list with field decoration).
   - Symptom in trace: client sends e.g. `queryObjects(11, name1, addr)`;
     reply contains only bare ids (`20001`, `20002`, ...), no
     `name1[..] addr[..]` decorations. JMRI's accessory list shows blank
     names/addresses.
   - Fix location: `parser.js` `handleSwMgr` queryObjects branch
     (lines 470–473). Today it's a single `ids.map(id => \`${id}\`)`.
     Mirror the field-include logic from `queryLoks` (lines 360–389)
     against `state.accessories[id].name1` / `.addr` / `.addrext`.

2. **`get(27, delay) / set(27, delay[..])`** (Booster).
   - Symptom in trace: client sends `get(27, delay)`; reply is
     `<REPLY get(27, delay)>\n<END 0 (OK)>` with no body line. Some
     throttles read this as "booster unreachable" and retry forever.
   - Fix location: `parser.js` DISPATCH table (line ~909). Today `booster`
     is routed to `handleStub`; replace with a dedicated handler that
     returns `27 delay[<n>]` for `get(27, delay)` and accepts/stores the
     value on `set(27, delay[..])`. State slot to add on `defaultState()`.

3. **`get(11, switch[<MOT|DCC><addr><r|g>])` before any matching
   `set`** (iTrain polling current accessory state at startup).
   - Symptom in trace: very early in a session, before any `set(11,
     switch[..])` has been issued, client sends `get(11, switch[DCC4g])`
     and gets `<END 19 (NERROR_NOOBJECT)>`. iTrain may show the
     accessory as "unknown state" indefinitely.
   - Fix location: `parser.js` `handleSwMgr` get branch, switch sub-branch
     (lines 434–444). Today it reads only from `state.accessoriesByAddr`,
     which is populated lazily on first set. Either seed
     `accessoriesByAddr` from `layout-map.json` at boot (preferred — give
     it a sensible default per-address based on HA's current state), or
     return `0 (OK)` with the queried position assumed to match (no — that
     would fabricate data; do not do that). Preferred fix is the layout-map
     seed.

4. **`set(<lokId>, funcdesc[n, <symbolic-code>])`** (JMRI sends a
   keyword like `LIGHT` or `HORN` where the spec wants a numeric code).
   - Symptom in trace: client sends e.g.
     `set(1001, funcdesc[0, LIGHT, moment])`; reply is
     `<END 0 (OK)>`, but a subsequent `get(1001, funcdesc[0])` returns
     `funcdesc[0, NaN, moment]`. Function description never displays
     correctly on the throttle.
   - Fix location: `parser.js` `setLokFields` funcdesc branch
     (lines 714–719). Today it does `Number(tokVal(o.args[1]))` which
     yields NaN for non-numeric tokens. Either accept the raw token
     (`tokVal(o.args[1])`) and store it as-is, or map JMRI's keyword set
     to the ESU numeric codes from spec §7.11.

Anything else that lands in the `unhandled` debug node (parser output 4)
or shows a `<END 15>` / `<END 19>` in the trace is a new gap — bring it
back here as a one-line dispatch-table addition.

## How to verify against live iTrain or JMRI

Once you've actually imported `flow.json` somewhere (BowmanMtn now or
Trail Controller Mac later):

1. In the Node-RED editor, find the `toggle trace log` inject node on
   the FaCoS tab. Click its button. The debug pane should show
   `FaCoS trace ENABLED`.
2. Point iTrain (or JMRI throttle, or Rocrail) at `tcp://<host>:15471`.
3. Drive the operations you actually care about: throw a turnout,
   start/stop a lok, change function bindings, etc.
4. Back on the Node-RED host:

```
tail -f cursor-scratch/facos-trace.log
```

Format per line:

```
2026-05-17T15:00:00.000Z<TAB>sessionId<TAB>"raw frame"<TAB>cmd:id
```

5. Anything that hits the `unhandled` debug node (Output 4 of the parser
   function) is a command the parser flagged as either a parse failure
   or a `NERROR_BADPARAMETER` / `NERROR_NOOBJECT` response. Each is a
   one-line dispatch-table addition or a new branch inside an existing
   handler. Capture the raw frame from the trace log and bring it back
   here for a fix.

6. Click `toggle trace log` again when you're done — `FaCoS trace
   DISABLED`. The log file is not rotated; it appends until you delete
   it. `.gitignore` already excludes `cursor-scratch/`.
